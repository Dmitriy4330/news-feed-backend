# News Feed API

Бэкенд для новостной ленты с подписками на пользователей.

## Функциональность

- Регистрация и аутентификация пользователей (JWT)
- Создание, редактирование, удаление постов
- Подписка/отписка на пользователей
- Лента новостей 

## Как запустить:

### 1. Клонирование репозитория
git clone https://github.com/ваш-логин/news-feed-backend.git
cd news-feed-backend
### 2. Запуск контейнеров
docker-compose up -d
### 3. Установка зависимостей
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
### 4. Запуск сервера
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
### 5. Документация API
Открываем в браузере: http://localhost:8000/docs

## API Endpoints
### Аутентификация
POST /auth/register - регистрация

POST /auth/login - вход

### Посты
POST /posts/ - создать пост

GET /posts/{id} - получить пост

PUT /posts/{id} - обновить пост

DELETE /posts/{id} - удалить пост

### Подписки
POST /subscriptions/{author_id} - подписаться

DELETE /subscriptions/{author_id} - отписаться

GET /subscriptions/my - мои подписки

### Лента
GET /feed/?limit=20&cursor=... - получить ленту


## CI/CD Pipeline с Blue-Green деплоем

В рамках лабораторной работы был настроен CI/CD пайплайн для автоматической сборки, тестирования и деплоя приложения на удалённую виртуальную машину.

Пайплайн реализован с использованием GitHub Actions и запускается при каждом push в ветку main.

### Общая логика пайплайна

CI/CD состоит из трёх последовательных этапов:

1. test
2. build-and-push
3. deploy

Каждый следующий этап запускается только при успешном завершении предыдущего.

### Этап test

На этом этапе выполняются автоматические тесты приложения:

- используется pytest;
- PostgreSQL и Redis поднимаются как сервисы GitHub Actions;
- проверяется корректность схем и базовой бизнес-логики;
- результаты тестов сохраняются как artifacts.

### Этап build-and-push

На этом этапе:

- собирается Docker-образ приложения;
- образ тегируется:
  - latest
  - хэшем коммита;
- образ публикуется в Docker Hub.

Каждый деплой всегда привязан к конкретному коммиту.
### Этап deploy (Blue-Green)

Деплой выполняется по SSH на удалённую виртуальную машину с использованием
Blue-Green.

Используется следующая схема:

- app_blue и app_green — две версии приложения;
- nginx — направляет трафик на активную версию;
- PostgreSQL и Redis являются общими для обеих версий.

Алгоритм деплоя:
1. Определяется неактивная версия (blue или green);
2. Для неё загружается новый Docker-образ;
3. Контейнер пересоздаётся с новым образом;
4. Выполняется health-check;
5. nginx переключает трафик на новую версию;
6. Старая версия останавливается.

Таким образом, обновление происходит без остановки сервиса.

### Локальный запуск
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d
# Проверка
curl http://localhost/health

# Переключение версий
./scripts/blue-green-switch.sh
